›SCP Deny-List + Acquired Account Carve-OutScenarios 1 · 6
IF / WHEN
Root SCP blocks a resource type AND a new/acquired account needs temporary access to that blocked resource
THEN
Create a temporary OU (e.g. Onboarding). Attach an Allow SCP for the carve-out to that OU. Move the account into it. After freeze period, move to Production OU — root SCP re-applies automatically.
NEVER
• IAM permission boundaries — cannot override SCPs
• Adding a second Deny SCP to child OU — deny lists stack, they do not cancel
• Condition keys on root SCP (e.g. aws:PrincipalAccount) — per-account carve-outs must use OU structure
WHY IT WORKS
Effective permissions = intersection of ALL SCPs in the OU hierarchy. A child-OU Allow SCP carves out the exception while the root SCP remains unchanged. Authorization order: Organization SCPs → Resource-based policies → IAM policies → Permission boundaries → Session policies.

›SCP Evaluation Logic — Full Authorization ChainGeneral — High Frequency
IF / WHEN
Question asks whether an action will be allowed or denied given a combination of SCPs, IAM policies, and resource policies
THEN — EVALUATION ORDER
StepCheckResult if Deny
1Explicit Deny anywhere (SCP, IAM, resource policy)DENY — final
2Organization SCP — does it allow the action?DENY if not allowed
3Resource-based policy (if cross-account) — explicit Allow?DENY if absent
4IAM identity policy — explicit Allow?DENY if absent
5Permission boundary (if exists) — within boundary?DENY if outside
6Session policy (if AssumeRole) — within session scope?DENY if outside
CRITICAL TRAPS
• Management account is exempt from SCPs — SCPs never apply to the management account
• SCPs do not grant permissions — they only restrict what IAM policies CAN grant
• NotAction in SCP = allow everything except listed actions (deny-list pattern)
• Action in SCP with Allow = allow-list pattern — everything else is implicitly denied
SCPsIAMAuthorization
›Merging Two AWS Organizations Post-AcquisitionScenario 46
IF / WHEN
Merging two AWS Organizations after acquisition — move accounts, apply SCPs, preserve IAM resources, activate consolidated billing, unify SSO
THEN
Remove member accounts from acquired Org → management account sends Organization invitations → accounts accept → SCPs apply immediately on join → consolidated billing activates automatically → onboard employees to existing IAM Identity Center (Okta) by provisioning in IdP.
NEVER
• "AWS Support can merge Organizations" — this capability does not exist
• Creating new accounts and migrating resources — violates no-re-provisioning requirement
• Cross-org consolidated billing — not a supported feature, billing only works within one Org
WHY IT WORKS
Joining an Organization does not touch intra-account IAM resources. Consolidated billing and SCP enforcement activate the moment an account joins.

›High-Throughput Streaming + Exactly-Once + Sub-200ms LatencyScenario 2

THEN
